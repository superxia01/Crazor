# Crazor 数据架构：知识库与数据库的边界

> 核心原则：**需要被数出来的进数据库，需要被读出来（当素材/当参考）的进知识库。**

---

## 划分原则

| | 知识库 (docs / md 文件) | 数据库 (SQLite) |
|---|---|---|
| **本质** | 文档、内容、策略、素材 | 记录、状态、数值、关联 |
| **特征** | 长文本、自由结构、需要AI阅读理解全文 | 结构化字段、需要筛选/排序/聚合/统计 |
| **读写方式** | AI 读取→理解→创作→写回 | MCP 工具一行调用、前端表格/看板展示 |
| **变更频率** | 内容随时改，格式灵活 | 字段相对固定，custom_data JSON 兜底扩展 |
| **典型问法** | "这个策略是什么""帮我写一篇关于X的文章" | "多少个客户在跟进中""本周公众号发了几条" |

---

## 客户管理 (CRM)

### 数据库 — 被数出来的

- 客户档案（姓名、公司、阶段、状态、评级、商机金额）
- 跟进记录（日期、方式、内容摘要、下一步）
- 交易记录（金额、日期、产品类型、回款状态）
- 渠道档案（名称、合作模式、评级、累计数据）
- 统计数据（漏斗、收入、逾期数）

前端需要：看板按阶段拖拽、筛选活跃客户、统计漏斗概览。

### 知识库 — 被读出来的

- 客户背景深度分析（长文，不适合塞数据库字段）
- 销售方法论、话术模板
- 行业分析笔记
- 成交案例复盘
- 跟进策略文档（什么阶段做什么动作）

通过 `contact_id` 关联知识库文档和客户记录。AI 打电话时读知识库获取背景，写跟进记录时存数据库。

## 内容生产

### 数据库 — 被数出来的（内容追踪表）

每一篇作品只存**追踪元数据**：

| 字段 | 说明 |
|------|------|
| title | 标题 |
| platform | 平台（公众号/小红书/抖音/视频号/知识星球/朋友圈） |
| form | 形式（文章/图文/口播稿/Talk） |
| status | 状态（选题中/草稿/拍摄中/待发布/已发布/关闭） |
| published_at | 发布日期 |
| topic_source | 选题来源 |
| doc_id | 关联知识库中的完整文档ID |
| views | 阅读量/播放量 |
| likes | 点赞 |
| comments | 评论数 |
| shares | 转发/收藏 |
| custom_data | JSON 扩展（粉丝增长、转化率等） |

前端需要：
- 看板按状态分列拖拽
- 筛选"本周公众号发了几条"
- 统计"小红书哪篇数据最好"
- 每日硬指标完成度检查

### 知识库 — 被读出来的

- **流水线设计文档** — AI 读它知道步骤怎么走
- **各平台运营策略** — 写稿时参考风格、调性
- **写作标准/模板** — 口播稿结构、图文规范
- **素材原文**（getbiji 笔记、AI 资讯、日常观察）— AI 从素材提炼选题
- **完整的文章/图文文案/口播稿正文** — 创作产出物本身
- **爆款标题模型、金句库、方法论** — 创作参考

---

## 核心关联机制

```
内容正文（文章/口播稿/图文文案）  →  知识库文档
内容追踪（标题/平台/状态/数据）    →  数据库记录

两者通过 doc_id 关联：
  数据库记录存 doc_id → 指向知识库中的完整文档
```

### 典型工作流

| 场景 | 数据流 |
|------|--------|
| Hermes 写稿 | 读知识库素材 → 创作 → 写入知识库文档 → MCP 创建数据库追踪记录（含 doc_id） |
| 前端展示 | DataView 列表/看板显示追踪数据，点击详情查看知识库中的完整文稿 |
| 数据回收 | MCP 更新数据库的 views/likes 字段，不改知识库文档 |
| 日常提问 | "这周发了多少条" → 查数据库；"帮我看看那篇文章写得怎么样" → 读知识库 |
