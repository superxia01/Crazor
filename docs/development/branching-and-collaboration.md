# 分支与多人协作规范

本文档用于约束 Crazor 后续多人协作开发，避免多人直接改 `master` 造成冲突和不可回滚变更。

## 分支模型

- `master`：稳定主线，只合并已经验证的功能。
- `feat/*`：新功能，例如 `feat/docker-deployment`、`feat/case-workspace`。
- `fix/*`：缺陷修复。
- `docs/*`：纯文档修改。
- `chore/*`：工程维护、依赖、脚本和配置。
- `experiment/*`：短期实验分支，不直接合并，必须整理后转为正式分支。

## 开发流程

1. 从最新 `master` 拉分支。
2. 一个分支只解决一个明确目标。
3. 提交前确认没有无关文件、临时文件和本地数据。
4. 修改前端、后端或部署配置时，必须补充对应文档。
5. 合并前至少完成本地启动或配置校验。

## 命名示例

```bash
git switch master
git pull --ff-only
git switch -c feat/case-workspace
```

## 提交粒度

建议提交按以下粒度拆分：

- 部署配置。
- 后端模型或 API。
- 前端页面。
- 文档。
- 测试或验证脚本。

避免把大段无关 UI、后端、文档和数据文件混在一个提交里。

## 禁止事项

- 禁止直接提交 `node_modules`、`dist`、`.env`、`data/` 和本地数据库。
- 禁止把 mock 数据混入真实运行数据。
- 禁止在文档、日志和示例中写入真实密钥。
- 禁止多人长期共用同一个功能分支。
