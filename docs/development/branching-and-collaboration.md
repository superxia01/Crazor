# 分支与多人协作规范

本文档用于约束 Crazor 后续多人协作开发，避免多人直接改 `master` 造成冲突和不可回滚变更。

## 分支模型

- `master`：稳定主线，只合并已经验证的功能。
- `feat/*`：新功能，例如 `feat/docker-deployment`、`feat/case-workspace`。
- `fix/*`：缺陷修复。
- `docs/*`：纯文档修改。
- `chore/*`：工程维护、依赖、脚本和配置。
- `experiment/*`：短期实验分支，不直接合并，必须整理后转为正式分支。

## 开发流程

1. 每次开始新任务前，先执行远端同步检查：`git fetch --all --prune`。
2. 对比当前工作分支、`origin/master` 和目标 PR 分支，确认是否有新增需求、架构调整或交付修正。
3. 如果远端有新增需求，先做评估记录：判断是否立即整合、拆成后续任务，或作为冲突风险暂缓。
4. 从最新 `master` 或明确目标基线拉分支。
5. 一个分支只解决一个明确目标。
6. 提交前确认没有无关文件、临时文件和本地数据。
7. 修改前端、后端或部署配置时，必须补充对应文档。
8. 合并前至少完成本地启动或配置校验。

## 任务启动检查

每次进入新的开发任务前，必须先完成以下检查，避免在旧上下文上继续堆功能：

```bash
git fetch --all --prune
git status -sb
git log --oneline --decorate --graph --all -n 12
git diff --name-status HEAD..origin/master
```

检查结论要回答三个问题：

- 是否有新增产品需求或架构目标。
- 是否有交付形态、认证、数据结构、API 或前端入口变更。
- 是否会影响当前分支，是否需要先合并、手工整合、拆分任务或记录为后续计划。

如果发现新增需求，不直接开始写代码。先给出简短评估，再把结论同步到对应入口文档或计划里。

## 命名示例

```bash
git switch master
git pull --ff-only
git switch -c feat/case-workspace
```

## 提交粒度

建议提交按以下粒度拆分：

- 部署配置。
- 后端模型或 API。
- 前端页面。
- 文档。
- 测试或验证脚本。

避免把大段无关 UI、后端、文档和数据文件混在一个提交里。

## 禁止事项

- 禁止直接提交 `node_modules`、`dist`、`.env`、`data/` 和本地数据库。
- 禁止把 mock 数据混入真实运行数据。
- 禁止在文档、日志和示例中写入真实密钥。
- 禁止多人长期共用同一个功能分支。
